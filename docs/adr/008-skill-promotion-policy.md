# ADR-008: Skill Promotion Policy

**Status:** Accepted
**Date:** 2026-04-17
**Deciders:** DreamFi Engineering

---

## Context

DreamFi uses an iterative prompt improvement loop (ADR-004) where each skill's prompt version is evaluated against locked binary criteria (ADR-003). The critical question is: what signal determines whether a new prompt version replaces the current production version?

The autoresearch toolkit principle is unambiguous: **score improvement is the only promotion signal.** A new prompt version is promoted to ACTIVE only if its evaluation round score exceeds the baseline (current ACTIVE version's score). No other factor -- human preference, time elapsed, stakeholder request, or urgency -- can promote a prompt version.

This principle prevents several failure modes:
- "Ship it, it looks good enough" (subjective quality that may not match eval criteria).
- "We need it by Friday" (time pressure overriding quality gates).
- "The VP prefers this version" (authority-based promotion that bypasses evaluation).
- "It has been running for a week with no complaints" (absence of evidence mistaken for evidence of quality).

---

## Decision

**Only score improvement promotes prompt versions. No manual override.**

### Promotion Rules

1. A prompt version is promoted from TESTING to ACTIVE if and only if:
   ```
   current_round.aggregate_score > baseline_round.aggregate_score
   ```
   Where `baseline_round` is the most recent KEPT round for the same skill using the same eval file version.

2. If the score is equal to or lower than the baseline, the prompt version is reverted:
   ```
   if current_round.aggregate_score <= baseline_round.aggregate_score:
       prompt_version.status = REVERTED
       round.outcome = REVERTED
   ```

3. There is no manual promotion path. No API endpoint, CLI command, or database operation allows bypassing the score check.

4. The score comparison is strictly numeric. No tolerance bands, no rounding, no "close enough."

5. For the first round of a new eval file version (no baseline exists), the round establishes the baseline. The current ACTIVE prompt version is evaluated under the new criteria to set the initial score. This is an evaluation round, not a promotion event.

### What Happens After Promotion

When a prompt version is promoted:

1. The previous ACTIVE version is marked SUPERSEDED.
2. The new version is marked ACTIVE.
3. Outputs from the promoting round that meet gold example criteria (all criteria pass, score >= 90%) are written to `gold_example_registry`.
4. The promoting round is logged with full provenance: prompt_version_id, eval_file_version, aggregate_score, baseline_score.
5. Any artifacts generated by the superseded version that have not yet been published are flagged for re-generation with the new ACTIVE version.

### What Happens After Reversion

When a prompt version is reverted:

1. The version is marked REVERTED.
2. The previous ACTIVE version remains ACTIVE.
3. The failure is logged to `skill_failure_patterns` with the change_description from the reverted version.
4. No outputs from the reverted round are eligible for gold example promotion.

## Alternatives Considered

### Human approval gate

**Pros:** A human reviewer approves or rejects each prompt version before production deployment. Captures quality dimensions that automated criteria may miss (tone nuance, brand alignment, strategic fit).

**Cons:** Creates a bottleneck. Reviewer availability limits iteration speed. Reviewers develop fatigue and rubber-stamp after initial novelty. Reviewer judgment may conflict with eval criteria, creating inconsistency. The autoresearch principle specifically eliminates human judgment from the promotion decision to enable rapid, consistent iteration.

**Rejected because:** Human input is valuable but belongs at a different point in the loop: defining eval criteria, writing test inputs, and reviewing gold examples. The promotion decision itself should be mechanical, based on the criteria that humans already defined.

### Time-based promotion (canary)

**Pros:** New prompt versions are deployed to a subset of users. If no regressions are detected within N days, the version is promoted. This mimics software deployment best practices (canary releases).

**Cons:** "No regressions detected" is a weak signal when detection depends on user complaints. In-app feedback loops for generated content quality are immature. The time window adds days of latency to every iteration. Small score regressions may go unnoticed for long periods.

**Rejected because:** Binary eval criteria provide faster, more reliable quality signals than user-reported regressions. Canary deployment is appropriate for software with monitoring, not for prompt versions evaluated against defined criteria.

### Score improvement + human approval (dual gate)

**Pros:** Combines automated scoring with human oversight. Score improvement is necessary but not sufficient; a human must also approve. Provides a safety net for cases where eval criteria are incomplete.

**Cons:** The human gate negates the speed advantage of automated evaluation. If eval criteria are incomplete, the correct fix is to improve the criteria (create a new eval file version), not to patch around them with human review. Dual gates add process complexity.

**Rejected because:** If the eval criteria are insufficient to determine quality, the criteria should be improved. Using human approval as a backstop for bad criteria masks the problem and delays fixing it.

### Threshold-based promotion (score >= X)

**Pros:** Promote any version that achieves a minimum score threshold (e.g., >= 0.85) regardless of baseline. Simple and understandable.

**Cons:** Does not prevent regressions. A prompt version scoring 0.86 could replace one scoring 0.92. The threshold sets a floor but not a direction. The autoresearch principle requires monotonic improvement, not just minimum quality.

**Rejected because:** The goal is continuous improvement, not just meeting a minimum bar. Score improvement over baseline ensures the system never moves backwards.

---

## Tradeoffs

**Accepted tradeoffs:**
- **No emergency override.** If a critical prompt change is needed (e.g., compliance language update) and the change causes a score regression, it cannot be promoted. The team must either fix the regression or update eval criteria to account for the new requirement.
- **Score gaming is possible.** In theory, a prompt could be engineered to pass binary criteria without actually being high quality. The criteria may not capture all quality dimensions.
- **Small improvements compound slowly.** Each round improves by a small margin. Transformative improvements that temporarily regress some criteria while dramatically improving others are blocked.

**Mitigated by:**
- Compliance and regulatory requirements should be encoded as eval criteria. If a new compliance requirement causes score regression, it means existing criteria conflict with the requirement and need updating (new eval file version).
- Score gaming is bounded by the quality of the eval criteria. Investing in comprehensive, well-designed criteria is the primary defense. Gold example review provides a secondary check.
- Transformative changes can be accomplished by creating a new eval file version with updated criteria, resetting the baseline, and iterating from the new starting point.

---

## Consequences

1. Prompt quality is monotonically non-decreasing within an eval file version. The system never deploys a prompt version that scored worse than its predecessor (under the same criteria).
2. The promotion decision is fully auditable. Every promotion is recorded with exact scores, prompt versions, and eval file versions.
3. Teams cannot bypass quality checks under time pressure. The only path to promotion is to make changes that actually improve scores.
4. Eval criteria quality becomes the most important investment. Bad criteria lead to prompt versions that pass criteria but fail in practice. Good criteria make the automated promotion loop trustworthy.
5. The `gold_example_registry` grows monotonically (new entries from every kept round). This provides an expanding corpus of high-quality outputs for few-shot prompting.
6. `skill_failure_patterns` captures every reversion, building a knowledge base of what types of prompt changes tend to fail for each skill.

---

## Rollback Plan

1. **Add a manual override with audit.** Introduce a `force_promote` operation that bypasses the score check. Require a written justification, log the override to an audit table, and alert the team. This weakens the policy but provides an escape hatch.
2. **Add tolerance band.** Instead of strict `>`, use `>= baseline - tolerance` where tolerance is a small value (e.g., 0.02). This allows promotions where the score drops insignificantly. Risks: gradual regression through accumulated tolerance.
3. **Add dual-gate for specific skills.** For high-risk skills (agent_system_prompt, support_agent), add a human approval gate in addition to score improvement. For lower-risk skills, keep score-only promotion. Risks: inconsistent policy across skills.
4. **Revert to manual promotion.** Disable automated promotion entirely. Prompt versions are promoted by human decision. Score is advisory. This is a full rollback of the policy and should be considered a last resort.
