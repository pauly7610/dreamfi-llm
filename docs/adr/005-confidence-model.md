# ADR-005: Confidence Model

**Status:** Accepted
**Date:** 2026-04-17
**Deciders:** DreamFi Engineering

---

## Context

Every artifact generated by the DreamFi platform -- system prompts, support responses, product descriptions, email copy -- needs a trust signal that downstream consumers (humans, publishing pipelines, dashboards) can use to decide whether to act on it. A single "confidence score" is more actionable than exposing raw eval scores, freshness timestamps, and citation counts separately.

The confidence score must reflect multiple quality dimensions:

- **Eval quality:** Did the generating skill's prompt version pass its evaluation criteria?
- **Source freshness:** Are the cited knowledge hub entities up to date?
- **Citation coverage:** Does the artifact cite verifiable sources?
- **Hard-gate status:** Are there any absolute blockers (failed eval, stale sources, skill mismatch)?

The score must be interpretable: a human reading "confidence: 0.82" should have a consistent understanding of what that means across all skills and artifact types.

---

## Decision

**Combine eval score, freshness, citation coverage, and hard-gate status into a single confidence score.** Hard-gate failures override the formula and force confidence to 0.0.

### Formula

```
confidence = (w1 * eval_score) + (w2 * freshness_score) + (w3 * citation_coverage) + (w4 * recency_bonus)

Where:
  w1 = 0.40  (eval quality weight)
  w2 = 0.30  (freshness weight)
  w3 = 0.20  (citation coverage weight)
  w4 = 0.10  (recency bonus weight)
```

### Component Definitions

**eval_score (0.0 - 1.0):**
The aggregate score from the most recent KEPT evaluation round for the skill that generated this artifact. Computed as the fraction of (output, criterion) pairs that passed.

**freshness_score (0.0 - 1.0):**
The minimum freshness_score across all source entities cited by this artifact. The weakest link determines freshness. If no sources are cited, freshness_score = 0.5 (neutral).

```
freshness_score = min(cited_entity.freshness_score for each cited entity)
```

**citation_coverage (0.0 - 1.0):**
The fraction of factual claims in the artifact that are backed by a citation to a canonical entity. Computed by the eval criteria (specific criterion for citation presence).

```
citation_coverage = (number of cited claims) / (total number of factual claims)
```

If the artifact has no factual claims (e.g., a headline), citation_coverage = 1.0.

**recency_bonus (0.0 - 1.0):**
Bonus for how recently the artifact was generated, decaying over time.

```
hours_since_generation = (now - artifact.created_at) in hours
recency_bonus = max(0.0, 1.0 - (hours_since_generation / 168))  // decays to 0 over 7 days
```

### Hard-Gate Override

If any of the following conditions are true, the formula is bypassed and confidence is forced to 0.0:

| Hard Gate                | Condition                                       |
|--------------------------|--------------------------------------------------|
| Eval failure             | Latest round for this skill has outcome = REVERTED |
| Source staleness          | Any cited entity has freshness_score < 0.3       |
| Skill/artifact mismatch | Artifact type does not match skill's declared output_type |

```
if eval_failed OR min_freshness < 0.3 OR skill_mismatch:
    confidence = 0.0
    publish_blocked = true
```

### Thresholds

| Confidence Range | Label       | Publish Allowed | Dashboard Color |
|------------------|-------------|-----------------|-----------------|
| 0.90 - 1.00     | High        | Yes             | Green           |
| 0.70 - 0.89     | Medium      | Yes             | Yellow          |
| 0.50 - 0.69     | Low         | Yes (flagged)   | Orange          |
| 0.00 - 0.49     | Untrustable | No              | Red             |

Artifacts with confidence in the "Low" range are publishable but carry a visible flag warning consumers that confidence is below the preferred threshold.

---

## Alternatives Considered

### Separate scores without aggregation

**Pros:** More transparent. Consumers can see exactly which dimension is weak. No information loss from aggregation.

**Cons:** Consumers must understand and evaluate four separate scores, which is cognitively expensive. Decision logic ("should I publish this?") must be reimplemented by every consumer. Different consumers may weigh dimensions differently, leading to inconsistent behavior.

**Rejected because:** A single actionable score with documented thresholds is more usable for automated publishing gates and human review workflows. The component scores remain available in the data model for detailed inspection.

### Binary pass/fail (no numeric score)

**Pros:** Simplest possible model. Either an artifact is trustworthy or it is not. No ambiguity.

**Cons:** Loses nuance. An artifact with eval_score = 0.95 and slightly stale sources (freshness = 0.35) is very different from an artifact with eval_score = 0.50 and fresh sources. Binary scoring collapses both to "fail." This makes it difficult to prioritize improvements and triage.

**Rejected because:** The confidence score serves both automated gates (which need binary thresholds, provided by the threshold table) and human review (which benefits from a continuous signal).

### Machine-learned confidence model

**Pros:** Could capture non-linear interactions between components. Could learn from human feedback on artifact quality.

**Cons:** Requires training data (human quality judgments on generated artifacts). Adds model training and serving infrastructure. Harder to interpret and debug. Overfitting risk with limited training data.

**Rejected because:** The linear weighted formula is interpretable, debuggable, and sufficient for the current use case. If quality signals prove non-linear, the formula can be upgraded to a learned model later without changing the interface.

---

## Tradeoffs

**Accepted tradeoffs:**
- **Weights are somewhat arbitrary.** The initial weights (0.4/0.3/0.2/0.1) reflect a judgment call about relative importance. They may need tuning as the system matures.
- **Min-freshness is conservative.** Taking the minimum freshness across all citations means one stale source drags down the entire score. This may be overly punitive for artifacts with many citations where only one is stale.
- **Hard gates are strict.** A single failed hard gate forces confidence to 0.0 regardless of other dimensions. This prevents publishing artifacts that might be mostly good.

**Mitigated by:**
- Weights are configurable per skill family. Phase 5 (UI copy) may weight citation_coverage lower since headlines have fewer factual claims.
- The freshness threshold for hard-gate (0.3) is set low enough that only severely stale sources trigger the block.
- Hard-gate strictness is intentional: it prevents publishing artifacts with known quality problems. The strict approach is safer and can be relaxed later.

---

## Consequences

1. Every generated artifact carries a `confidence` field (float, 0.0-1.0) and a `confidence_label` (High/Medium/Low/Untrustable).
2. The publishing pipeline checks `confidence >= 0.50` (configurable threshold) before allowing publish.
3. Dashboards display confidence as a color-coded indicator, giving humans quick quality visibility.
4. Component scores (eval_score, freshness_score, citation_coverage, recency_bonus) are stored alongside the aggregate for auditability.
5. Weight tuning is a configuration change, not a code change. Adjustments can be made per skill family.
6. The confidence formula is deterministic and reproducible: given the same inputs, it always produces the same score.

---

## Rollback Plan

1. If the composite score proves misleading, fall back to exposing individual component scores to consumers. The aggregate score becomes advisory-only.
2. If hard gates are too strict, they can be converted to soft gates that apply a penalty (e.g., multiply confidence by 0.5 instead of forcing to 0.0) via configuration.
3. If weights need significant rework, a new version of the confidence model can be deployed alongside the old one (dual-write) for comparison before cutover.
4. Component scores are always stored independently, so no data is lost if the aggregation formula changes.
